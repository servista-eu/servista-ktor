Implement the core wiring layer in the `eu.servista.ktor` package. These four files form the backbone that Task 2b's supporting infrastructure files depend on.

**ServistaConfig.kt** (`eu.servista.ktor`):
- Data classes mirroring the `servista {}` HOCON block: `ServistaConfig`, `DatabaseConfig`, `KafkaConfig`, `KafkaConsumerConfig`, `LoggingConfig`, `RequestLoggingConfig`, `HealthConfig`
- `DatabaseConfig` and `KafkaConfig` are nullable (not all services need both)
- `KafkaConfig` contains:
  - `brokers: String` -- bootstrap.servers
  - `schemaRegistryUrl: String` -- Confluent Schema Registry URL
  - `consumer: KafkaConsumerConfig?` -- nullable, only needed by services that consume events
- `KafkaConsumerConfig` contains:
  - `groupId: String` -- consumer group ID (unique per service)
  - `autoOffsetReset: String = "earliest"` -- offset reset policy (earliest/latest)
- `ServistaConfig.load(config: ApplicationConfig)` companion function reads from Ktor's HOCON config
- `RequestLoggingConfig` has: `enabled: Boolean = true`, `logLevel: Level = Level.INFO`, `excludePaths: List<String> = listOf("/health")`
- Use `config.propertyOrNull()` for optional sections, `config.property()` for required ones

**StatusPagesSetup.kt** (`eu.servista.ktor.error`):
- `Application.installStatusPages()` extension function
- Exception handler for `ServistaException`: calls `cause.toProblemDetail()`, sets `Content-Type: application/problem+json`, responds with appropriate HTTP status code from `cause.statusCode`
- Catch-all handler for `Throwable`: logs error, responds with 500 and generic ProblemDetail with `type = "urn:servista:error:internal:unhandled"`, `title = "Internal Server Error"`

**ContentNegotiationSetup.kt** (`eu.servista.ktor.serialization`):
- `Application.installContentNegotiation()` extension function
- Register kotlinx.serialization JSON with `ignoreUnknownKeys = true`, `encodeDefaults = true`
- Register for both `application/json` and `application/problem+json` content types

**ServistaApplication.kt** (`eu.servista.ktor`):
- `Application.installServista()` extension function (no parameters, loads config from HOCON)
- `Application.installServista(config: ServistaConfig)` overload for testing
- Installation order (CRITICAL -- ContentNegotiation MUST be before StatusPages):
  1. ContentNegotiation (JSON serialization)
  2. StatusPages (error handling, needs serializer)
  3. CallLogging (request logging with MDC)
  4. MicrometerMetrics (metrics)
  5. Context interceptor (header extraction)
  6. Koin (DI) -- install with servistaModule
  7. Health routes (via routing)
  8. Metrics route (/metrics)
- The no-arg version calls `ServistaConfig.load(environment.config)` then delegates to the config overload
- Database and Kafka wiring are added in Plan 02

Implement the supporting infrastructure files that plug into the core wiring from Task 2a. Each concern gets its own package and setup file.

**HealthRoutes.kt** (`eu.servista.ktor.health`):
- `Route.healthRoutes(healthRegistry: HealthRegistry)` extension function
- Three endpoints: `/health/startup`, `/health/live`, `/health/ready`
- Each calls corresponding `healthRegistry.check*()` method
- Maps `HealthStatus.UP` and `DEGRADED` to 200 OK, `HealthStatus.DOWN` to 503 Service Unavailable
- Response body is the `HealthResponse` object (serialized by ContentNegotiation)

**RequestLoggingSetup.kt** (`eu.servista.ktor.logging`):
- `Application.installRequestLogging(config: RequestLoggingConfig)` extension function
- Only installs if `config.enabled` is true
- Filters out requests matching `config.excludePaths` patterns (simple prefix matching, strip trailing `*`)
- Adds MDC entries: `correlation_id` from `X-Correlation-Id` header, `organization_id` from `X-Organization-Id`, `account_id` from `X-Account-Id`
- Log level from config

**ContextInterceptor.kt** (`eu.servista.ktor.context`):
- `Application.installContextInterceptor()` extension function
- Intercepts at `ApplicationCallPipeline.Plugins` phase
- Extracts `X-Correlation-Id` (generates UUID if missing), `X-Organization-Id` (toLongOrNull), `X-Account-Id` (toLongOrNull)
- Creates `ServistaContext` coroutine context element from servista-commons
- Wraps `proceed()` in `withContext(ctx)` so all downstream handlers inherit the context

**MetricsSetup.kt** (`eu.servista.ktor.metrics`):
- `Application.installMetrics()` extension function
- Creates `PrometheusMeterRegistry(PrometheusConfig.DEFAULT)`
- Installs `MicrometerMetrics` plugin with that registry
- Registers a `/metrics` route that returns the Prometheus scrape output

**Logback configurations:**

`logback.xml` -- Routes to JSON or console config based on `SERVISTA_LOG_FORMAT` env var:
- Uses Logback conditional processing with `<if>` tag (requires Janino dependency -- add `org.codehaus.janino:janino` to build.gradle.kts if needed)
- If `SERVISTA_LOG_FORMAT` equals "json" -> include `logback-json.xml`
- Else -> include `logback-console.xml`
- Alternative if Janino causes issues: use `logback.configurationFile` system property approach

`logback-json.xml` -- Production JSON logging:
- Uses `LogstashEncoder` from logstash-logback-encoder
- Include MDC keys: `correlation_id`, `organization_id`, `account_id`, `trace_id`, `span_id`
- Custom fields: `service_name` from `${OTEL_SERVICE_NAME:-unknown}`, `service_version` from `${SERVICE_VERSION:-unknown}`
- Root logger level: INFO
- Appender: ConsoleAppender to stdout

`logback-console.xml` -- Development console logging:
- Standard PatternLayout with colored output: `%d{HH:mm:ss.SSS} %highlight(%-5level) [%thread] %cyan(%logger{36}) - %msg [%mdc]%n`
- Root logger level: DEBUG
- Appender: ConsoleAppender to stdout
